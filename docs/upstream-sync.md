# Upstream Sync Procedure

The evolved overlay stays aligned with `NousResearch/hermes-agent`. Upstream is
the primary product line; the evolved repository is an operational layer.

This workflow is upstream-first. When `upstream/main` conflicts with local fork
implementation, upstream supersedes. Take the official version first, then
back-build any deployment gaps as small overlay changes on top of it.

Current baseline:

- local `main` is synced to `upstream/main` as of merge commit `8126a3bb`
- `origin/main` and `gitlab/main` match that baseline
- future syncs start from this merged baseline, not from the retired
  `self-improve/upstream-sync-20260423` branch

The intake rule is:

- assume we are starting fresh from today's `NousResearch/hermes-agent`
- use upstream's implementation as the baseline whenever it touches the same
  surface as fork code
- delete local implementations that upstream now owns, even if that temporarily
  breaks Snapetech-only behavior
- backfill required Snapetech behavior after the upstream version is in place,
  with the smallest possible overlay and tests that describe the remaining gap

The invariant is non-negotiable:

- all required Snapetech outputs, tooling, connections, operators paths,
  pod behavior, and deployment integrations must still work after the sync
- or the newer Hermes functionality must replace them completely enough that
  the custom change is genuinely obsolete

If neither condition is true, the sync is not done.

## Remote Layout

Recommended remotes in the private package repo:

```bash
git remote add upstream https://github.com/NousResearch/hermes-agent.git
git remote set-url --push upstream DISABLED
git remote add origin git@github.com:<your-org>/<private-package-repo>.git
```

The public evolved mirror should be generated by `scripts/publish_evolved_repo.sh`
instead of manually merging private history into the public repository.

## Drift Report

Generate a report before deciding whether to merge upstream:

```bash
PRIVATE_REMOTE=origin \
PRIVATE_REF=main \
UPSTREAM_URL=https://github.com/NousResearch/hermes-agent.git \
UPSTREAM_REF=main \
REPORT_PATH=docs/upstream-sync-report.md \
scripts/upstream_sync_report.sh
```

Read:

- private-only commits
- upstream-only commits
- file impact summary
- files changed on both sides
- design-sensitive overlap
- keep-local-method overlap

The generated report is policy-aware. It should be treated as the first-pass
impact document for each sync attempt, not as permission to keep fork code ahead
of upstream. Its job is to identify what will break when upstream supersedes
local implementation, so the backfill can be deliberate.

Pay special attention to upstream changes touching:

- `run_agent.py`
- `gateway/run.py`
- `model_tools.py`
- `agent/context_compressor.py`
- `agent/model_metadata.py`
- `hermes_cli/config.py`
- `tools/approval.py`
- `tools/delegate_tool.py`
- `tools/mcp_tool.py`
- `deploy/`, `Dockerfile`, or installer scripts

These local deployment surfaces are high-risk and require targeted validation
after upstream supersedes them. They are not keep-local-by-default during a
merge:

- `gateway/run.py`
- `gateway/status.py`
- `hermes_cli/gateway.py`
- `agent/prompt_builder.py`
- `tools/skills_sync.py`
- `deploy/k8s/`
- pod/operator skills and workflow docs

Inherited upstream CI/workflow files should also be accepted from upstream
during the merge. If they are wrong for the private fork, remove or gate them in
a follow-up overlay commit with a clear reason:

- `.github/workflows/contributor-check.yml`
- `.github/workflows/deploy-site.yml`
- `.github/workflows/docker-publish.yml`
- `.github/workflows/docs-site-checks.yml`
- `.github/workflows/nix-lockfile-check.yml`
- `.github/workflows/nix-lockfile-fix.yml`
- `.github/workflows/nix.yml`
- `.github/workflows/skills-index.yml`
- `.github/workflows/supply-chain-audit.yml`
- `.github/actions/nix-setup/action.yml`

Do not silently prune upstream workflow files as a conflict-resolution shortcut.
The fork overlay may still gate private CI after the upstream merge is complete.

Current canonical rationale is documented in
[`docs/upstream-sync-design-deltas.md`](upstream-sync-design-deltas.md).

## Merge Checklist

1. Fetch upstream.
2. Generate the sync report.
3. Read upstream release notes or relevant PRs when available.
4. Classify overlap using the upstream-first test:
   - `upstream`: take upstream exactly
   - `upstream-plus-overlay`: take upstream, then add the smallest fork overlay
     needed to preserve required Snapetech behavior
   - `deferred-gap`: take upstream and record the broken local behavior for
     later backfill
5. If `git rev-list --left-right --count upstream/main...HEAD` already shows
   `0` upstream-only commits, refresh the report and design ledger instead of
   creating a new sync branch.
6. Otherwise create a branch from the private fork `main`. Do not work on
   `main` unless the operator explicitly asked for a direct sync on `main`.
7. Update the design-delta doc for the current sync attempt:
   - where upstream replaced local implementation
   - what Snapetech behavior needed an overlay after taking upstream
   - what remains broken or deferred after upstream adoption
8. Prefer merging upstream first. Cherry-picks are only for emergency hotfixes
   when a full upstream sync is explicitly out of scope.
9. Use `scripts/upstream_sync_prepare_branch.sh` only to prepare the branch and
   report by default. It does not attempt a merge unless `APPLY_MERGE=1`.
10. If you do run a merge, resolve conflicts by taking upstream first. Backfill
   deployment overlay intent only after the official code is in the tree.
11. Re-run targeted tests for touched surfaces.
12. Run `scripts/run_tests.sh` before pushing when code changed.
13. Rebuild the derived image if runtime files, TUI assets, or dependencies
   changed.
14. Run the public prereq and smoke scripts against a test deployment.
15. Publish the sanitized evolved mirror only after scans pass.

## Branch Preparation

Generate the branch and the report without merging:

```bash
REPORT_PATH=docs/upstream-sync-report.md \
scripts/upstream_sync_prepare_branch.sh
```

Attempt a merge after reviewing the impact report:

```bash
APPLY_MERGE=1 scripts/upstream_sync_prepare_branch.sh
```

The default behavior is deliberate. Branch-and-report is safe to automate.
When a merge is requested, upstream conflict resolutions are the default, and
fork-specific repairs happen afterward.

## When Already Synced

If `upstream/main...HEAD` shows `0` upstream-only commits:

1. refresh `docs/upstream-sync-report.md` so the ledger reflects the new
   baseline instead of stale triage numbers
2. update `docs/upstream-sync-design-deltas.md` for any local choices that
   survived the merge
3. record which previous local divergences were replaced by upstream
4. keep the report/script/skill current so the next sync starts from the real
   merged baseline

## Design Delta Ledger

Each upstream sync should leave behind durable notes in
[`docs/upstream-sync-design-deltas.md`](upstream-sync-design-deltas.md).

That ledger should answer:

- Which local overlays remain on top of upstream?
- Which upstream fixes did we adopt and why?
- Which local choices were dropped because upstream superseded them?
- Which Snapetech gaps are still deferred after taking upstream?

If a future operator or agent cannot tell why a local divergence exists, the
sync workflow is incomplete. If they cannot also tell whether that divergence
still survives the fresh-upstream test, the sync workflow is still incomplete.

## What To Upstream

This section is for deliberate human-maintained upstream contribution work. It
is not part of Hermes autonomous self-improvement.

Hermes self-improvement PRs and issues must target `example-org/hermes-agent-private`
only. Do not open self-improvement PRs or issues in
`NousResearch/hermes-agent`, and do not open them in
`snapetech/hermes-agent-evolved`.

When a human explicitly decides that a general fix should be contributed
upstream, candidates may include:

- provider registration and model metadata fixes
- generic context overflow handling
- gateway interruption and streaming fixes
- profile-safe path fixes
- tests that do not depend on private infrastructure
- docs that help non-cluster users

Keep deployment-local:

- private hostnames and service IPs
- self-hosted runner labels
- Discord channel/user IDs
- host SSH wrapper contracts
- live Kubernetes manifests with private assumptions
- model flags specific to one machine

## Publication

Dry-run the public mirror:

```bash
scripts/publish_evolved_repo.sh
```

Push only from a clean private tree:

```bash
scripts/publish_evolved_repo.sh --push
```

The publisher should include:

- public deployment guide
- evolved decisions
- reproducibility audit
- public examples
- prereq and smoke helpers

The publisher must not include:

- private live manifests
- credentials
- internal hostnames
- private IPs
- Discord IDs
- SSH key names
- inherited upstream GitHub Actions workflows

Inherited upstream workflows are intentionally stripped from the public mirror.
The mirror is a sanitized publication artifact, not the live deployment repo.
Add public workflows only when they are purpose-built for the sanitized overlay
and do not depend on private runners, private manifests, or repository secrets.
