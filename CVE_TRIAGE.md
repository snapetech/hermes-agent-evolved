# CVE Triage

Operator-tracked decisions for findings surfaced by
`scripts/stack_cve_check.py` and `docs/stack-cve-report.md`.

The generated report is regenerated by cron; this file is the human decision
ledger. Every finding in the latest report should end up with one of the
statuses below, or be explicitly noted as "deferred until next pass" with a
date.

## Status values

- `accept` — risk understood, no action needed (include reason).
- `upgrade-planned` — pending a dependency bump (include target + owner).
- `upgrade-done` — fixed, reference commit + removal from next report.
- `waived` — cannot fix (archived upstream, unfixable); describe mitigation.
- `pending` — not yet reviewed (remove when triaged).

### website/ (15 high, 23 moderate — all build-time / dev-only chain; `accept`)

**Investigation on 2026-04-23**:

1. `npm audit fix` (non-breaking) was attempted — it reduced the count to 25
   but **broke the build** because it bumped some `@docusaurus/*` packages
   while leaving others at 3.9.2 (Docusaurus enforces version lockstep
   across all official `@docusaurus/*` packages). Reverted.
2. Coordinated bump of every `@docusaurus/*` to 3.10.0 was attempted — build
   stayed green, but **audit count unchanged (still 38)** because the
   vulnerabilities are in transitive dev/build chain deps, not in docusaurus
   itself. Reverted to avoid a 954-line lockfile noise commit with zero
   security value.
3. `npm audit fix --force` would downgrade docusaurus to 2.2.0 (major
   regression from 3.9.2) and introduce React 19 peer-dep conflicts. Not
   viable.

**Classification of all 38**: every single advisory's root is a package
that lives only in the build toolchain or `npm run start` dev-server —
none ship in the built static site served by GitHub Pages.

| Package group | Count | Scope | Status |
|---|---:|---|---|
| `@docusaurus/*` (bundler, core, plugins, preset-classic, theme-\*) | 19 | Build-time SSR + compile step; compiled output is static HTML | `accept` |
| `@chevrotain/*` + `chevrotain` + `chevrotain-allstar` + `langium` | 4 | Mermaid parser build chain | `accept` |
| `mermaid` + `dompurify` | 2 | Mermaid diagram renderer (builds SVG at compile time) | `accept` |
| `webpack-dev-server` + `sockjs` + `uuid` (via sockjs/mermaid) | 3 | Dev-server only (`npm run start`) | `accept` |
| `copy-webpack-plugin` + `css-minimizer-webpack-plugin` + `serialize-javascript` | 3 | Build-chain webpack plugins | `accept` |
| `lodash` + `lodash-es` + `picomatch` + `brace-expansion` + `path-to-regexp` + `follow-redirects` + `undici` + `@easyops-cn/docusaurus-search-local` + misc | 7 | All transitive through build/dev-only chains above | `accept` |

**Reason for bulk `accept`**: the attack surface requires an attacker to
either (a) inject code into the build pipeline, which would require
compromising GitHub Actions or the self-hosted runner, or (b) exploit a
dev-server running locally during `npm run start`, which never runs in
production. Neither path is addressable by package upgrades; both are
addressed by the existing runner hardening and the principle that local
dev machines aren't exposed.

**Reopen criteria**: reopen individual rows if a Docusaurus minor/patch
release later resolves specific transitive advisories without downgrading
major versions, or if any of these packages later become runtime
dependencies (e.g. if Docusaurus changes to ship `dompurify` in the built
client bundle).
